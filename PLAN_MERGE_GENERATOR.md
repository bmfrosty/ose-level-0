## Key Design Decisions

### 1. Two URL parameter schemes

The merged `generator.html` will support two URL parameter formats:

#### A. Full/readable params — for bookmarking, sharing a *configuration* (not a character)
All generator settings serialized as named query-string parameters.  These pre-fill the
controls but do **not** embed the generated character itself.  The page generates fresh
on load (unless `autoGenerateOnLoad=0`).

| Param | Values | Notes |
|-------|--------|-------|
| `mode` | `basic` \| `advanced` | ← **new** (merged page only) |
| `p` | `ose` \| `smooth` \| `ll` | progression mode |
| `l` | `0`–`14` | starting level |
| `c` | class name | e.g. `Fighter` |
| `r` | race name | Advanced only, e.g. `Elf` |
| `dl` | `standard` \| `extended` | demihumanLimits, Basic only |
| `rcm` | `strict` \| `strict-human` \| `traditional-extended` \| `allow-all` | Advanced only |
| `prm` | `0` \| `9` \| `13` | primeRequisiteMode |
| `hc` | `0` \| `1` | healthyCharacters |
| `il` | `0` \| `1` | includeLevel0HP |
| `un` | `0` \| `1` | showUndeadNames |
| `ao` | `0` \| `1` | abilityOrdering (1=1977/OSE, 0=modern) |
| `qr` | `0` \| `1` | showQRCode |
| `wp` | `0`–`100` | wealthPct |
| `agl` | `0` \| `1` | autoGenerateOnLevelChange |
| `agc` | `0` \| `1` | autoGenerateOnClassChange |
| `ago` | `0` \| `1` | autoGenerateOnLoad |
| `ot` | `0` \| `1` | openInNewTab |
| `ap` | `0` \| `1` | autoPrintInNewTab |
| `n` | string | character name pre-fill |
| `s` | `3,3,3,3,3,3` | minimum ability scores (STR,INT,WIS,DEX,CON,CHA — 1977 order) |
| `zr` | race code | selectedRaceForZero (0-level race pre-select) |

The mode radio + `replaceState` keeps the URL in sync as the user changes settings.

#### B. Compact params (`?d=…`) — for QR codes / sharing a *generated character*
The existing `cp` object scheme in `shared-compact-codes.js`, gzip+base64url encoded.
Used by `charactersheet.html`.  See the field reference in that file.  The merged
`generator.html` will produce these exactly as `basic-ui.js` and `advanced-ui.js` do today.

### 3. Settings key — single `'generator'` key ✅ IMPLEMENTED

`generator-ui.js` uses a single localStorage key `'generator'` (via `getSettingsKey()`)
for all settings.  The single object stores all state including `mode`, `demihumanLimits`,
`raceClassMode`, `selectedRace`, and both modes' selections.  Switching modes calls
`applySettings(loadSettings('generator'))` which restores whichever fields are applicable.

This replaced the earlier plan to keep separate `'basic'` and `'advanced'` keys.
Benefits:
- Simpler `saveCurrentSettings()` — one unconditional `saveSettings()` call
- Mode switch restores the last-used selections for both modes from the same object
- No migration code needed going forward

Note: users with old `ose_settings_basic` or `ose_settings_advanced` localStorage entries
will simply start fresh with `ose_settings_generator` (no migration, settings reset to defaults).

### 4. Prime Requisite mode in Basic mode
`basic-ui.js` already has `primeRequisiteMode` — it's in Section 3 (Options).
It is a shared setting, not advanced-only. Keep in shared state.

### 5. Backwards compat: keep `m:'B'` and `m:'A'` cp codes
The cp object mode codes don't change — `basic-character-gen.js` still produces `m:'B'`
and `advanced-character-gen.js` still produces `m:'A'`. `charactersheet.html` already
decodes both. No change needed there.

### 6. `#demihumanLimitSection` not greyed at level 0
Unlike `#classSection` and `#raceClassSection`, the Demihuman Level Limits section is
intentionally **not** greyed out at level 0.  The setting must remain accessible at level 0
so the Standard/Extended choice carries through to the generated character sheet.
